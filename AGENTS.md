> 本文件是本项目的**统一执行规范**（适用于人类工程师与 AI 智能体）。  
> 目标：让任何新加入的成员在最短时间内明确“做什么、怎么做、做到什么程度算通过”，并避免常见返工与偏航。

## 1. 项目概述与边界

AI Second Brain 是一个基于 RAG 的个人知识库问答系统。核心体验来自 PRD：

- Chat-first 对话体验（三栏：左导航/中对话/右抽屉预览）
- 文件/文件夹管理与上传（含大文件分片与进度）
- 严格引用溯源：点击引用角标可跳转到原文对应位置并高亮
- 支持范围问答（global/folder/file scope）
- 低幻觉：仅基于检索上下文回答，找不到即明确告知

范围与阶段（以 PRD 为准）：

- MVP（V1.0）：TXT/PDF/Word 上传解析 + RAG 问答 + SSE 流式 + 引用溯源 + 基础文件/文件夹 CRUD
- V1.5：自动标签/摘要、范围问答 UI、对话历史
- V2.0：多模态、本地化、第三方同步、分享与更细权限

禁止事项：

- 不得实现 PRD 未提出或未规划到当前版本的功能并“顺手上线”。
- 不得修改“引用必须可溯源”的硬约束为“可选/尽量”。

## 2. 必须严格遵循的规则体系（硬约束）

### 2.1 需求与设计一致性

- 任何实现必须可在 PRD 中找到对应条目。
- 任何接口/数据结构变更必须同步更新：
  - `docs/API.md`（接口）
  - `docs/TECH_DESIGN.md`（架构/流程/模型/性能口径/安全/目录结构）
  - `docs/database.md`（数据库表结构与索引）

### 2.2 Postgres MVP（已决策）

- MVP 元数据存储统一使用 PostgreSQL（不再使用 SQLite）。
- 任何“为方便改回 SQLite”的变更必须先更新 `docs/TECH_DESIGN.md` 并说明迁移策略，否则不接受。

### 2.3 引用溯源（Citation）硬约束

- 前端体验要求：点击引用角标能打开右侧抽屉并定位/高亮原文位置。
- 输出质量要求：**每个事实句/要点必须包含引用标记**（不是只在答案末尾给一个 citations 列表）。
- 服务端必须做引用完整性校验：无法逐句绑定引用时必须降级（或引导用户缩小范围），不得“编造/补全”。

### 2.4 性能口径（必须可测）

- 问答 TTFB：< 2s（SSE 首条 `chunk` 发出）
- 10MB 文本/PDF 建库：< 15s（从 upload init 到 file ready 的全链路口径）
- 100MB 大文件：允许异步耗时更长，但必须提供进度并保证系统稳定不崩溃

### 2.5 安全与隐私（生命线）

- 云端：传输加密（TLS），敏感字段加密/哈希，访问鉴权与限流
- 本地：离线优先，不外传数据；提供一键擦除能力（规划项）
- 日志：默认不记录用户原文内容；如需排障必须具备脱敏/开关/采样策略（实现前需在 TDD 中落规则）

## 3. 文档体系（必须维护的配套文档）

### 3.1 `docs/TECH_DESIGN.md`（技术设计文档 / TDD）

用途：

- 端到端技术方案：架构、模块、数据模型、接口、数据流、安全、性能、测试计划、目录结构等。

更新触发器：

- 目录结构调整（例如按 fastapi-templates 重排）
- 组件替换（数据库/向量库/队列/对象存储）
- 引用协议、性能口径、安全策略的变更

### 3.2 `docs/API.md`（接口）

用途：

- 前后端联调、测试用的“单一事实来源”。

更新触发器：

- 任一 endpoint 的路径/方法/请求体/响应体变化
- SSE 事件协议变化（`chunk/citation/done` 等）
- 错误码、鉴权、分页、过滤参数变化

### 3.3 `docs/fronted/FRONTEND_TECHNICAL_DOCUMENT.md`（前端开发协作）

用途：

- 统一前端技术栈约定、目录结构、状态管理方案、组件设计规范、SSE 处理策略、引用点击预览策略等。

更新触发器：

- 引用点击预览的定位字段变化（page/start/end vs locator）
- scope UI/路由/布局形态变化
- 前端工程初始化后的真实脚本/命令变化

### 3.4 技术设计的“单一事实来源”

- 一切技术设计内容以 `docs/TECH_DESIGN.md` 为准，接口以 `docs/API.md` 为准。
 - 数据库表结构以 `docs/database.md` 为准。

## 4. 开发流程与协作规范

### 4.1 分支与提交

- 分支命名：`feat/`、`fix/`、`docs/`
- 提交信息：Conventional Commits（`feat:` / `fix:` / `docs:` / `refactor:`）

### 4.2 代码评审（必需）

- PR 需至少 1 位 reviewer（人类或具备相同标准的 AI 审核流程）
- 审核重点顺序：
  1. 是否符合 PRD/TDD/ADR
  2. 是否破坏引用溯源/性能口径/安全约束
  3. 是否有充分测试
  4. 是否引入不必要复杂度

### 4.3 变更最小化原则

- 只做与需求相关的改动；避免无关格式化/大范围重排。
- 调整结构时，先保证可运行与可导入，再逐步迁移实现。

## 5. 测试策略（完整）与验收标准

> 目标：任何成员都能回答“这个改动如何证明没坏”“上线前需要跑什么”“验收看哪些指标”。

### 5.1 测试金字塔

- 单元测试（70%）：纯函数/校验器/解析器/分块器/引用解析与校验
- 集成测试（20%）：API + DB + 向量库的关键链路
- E2E（10%）：真实页面操作（SSE 流式、引用点击打开抽屉并定位高亮）

### 5.2 后端测试（必须覆盖）

最小必测用例（MVP）：

- Auth：注册/登录（含错误分支）
- Upload：直接上传、小文件解析任务创建；分片上传 init/chunk/complete
- Tasks：进度接口返回结构稳定
- Chat（SSE）：事件顺序正确（`chunk* -> citation? -> done`），断线/异常能优雅结束
- CitationValidator：逐句引用校验与降级逻辑（这是核心硬约束）

验收标准（后端）：

- `import app` 不报错（可启动）
- API 结构与 `docs/API.md` 一致
- 核心链路的集成测试通过

### 5.3 前端测试（必须覆盖）

最小必测用例（MVP）：

- SSE 渲染：逐步输出可见，不卡顿；done 后状态收敛
- 引用角标：可点击，右侧抽屉弹出，定位与高亮可见（至少页码准确）
- 文件上传：状态变化与进度展示稳定（轮询或 WS）
- scope：global/folder/file 的选择影响对话创建与请求

验收标准（前端）：

- 三栏布局满足 PRD（信息不重叠、可用、可扫描）
- 引用溯源功能完整可用（点击跳转 + 高亮）

### 5.4 性能验收（必须可量化）

必须提供：

- TTFB 统计口径与监控/埋点
- 10MB 建库全链路耗时统计口径

验收阈值（来自 PRD/TDD）：

- TTFB < 2s
- 10MB 建库 < 15s（全链路口径）

## 6. 实施过程关键注意事项

### 6.1 SSE 在代理环境的坑

- 代理缓冲会导致“看似流式，实际一次性返回”。
- 必须确保相关 header 与网关配置（例如禁用 Nginx buffering）。

### 6.2 PDF 定位高亮的不确定性

- 文本抽取与渲染坐标可能不一致。
- MVP 优先保证“页码正确 + 高亮近似可见”，再逐步增强定位精度。

### 6.3 分片上传完整性

- 必须有 chunk 校验与最终文件校验（sha256/etag 等策略，落地前需在 TDD/API 写清楚）。
- 否则会出现“上传成功但内容损坏”的隐性问题。

### 6.4 数据删除的一致性

- 文件删除必须级联清理：DB chunks/tags/tasks + 向量库记录 + 对象存储 + 缓存。
- 不允许出现“原文删了但还能检索到”的状态。

## 7. 潜在风险提示（必须提前意识到）

- 成本风险：LLM 调用成本随 MAU/上下文长度快速增长，需要缓存与模型路由策略
- 质量风险：复杂 PDF 解析质量不稳定，需失败告警与降级策略
- 体验风险：引用逐句绑定要求极高，必须在协议与校验层“工程化落地”
- 兼容性风险：SSE、PDF 预览与高亮在不同浏览器/代理环境差异明显
- 本地部署风险：用户硬件差异导致体验分化，需要清晰的最低配置与降级方案

## 8. AI 智能体工作方式（建议执行顺序）

当你要开始一个任务时：

1. 先在 PRD/TDD/API/ADR 中定位需求与约束
2. 明确改动的“影响面”（接口/模型/前端协议/性能口径）
3. 先写/补测试，再实现（或至少同时补齐最小验证）
4. 改完后更新文档，使其成为下一位成员的“单一事实来源”
